# Phase 14 Plan 01: Make Time Estimate Field Mandatory

<objective>
Client requires the time estimate field to be mandatory for every item/service when creating an order. Without time estimates, workload planning and daily scheduling don't function properly.
</objective>

## Context

- Audrey quote: "Est-ce qu'il serait oblige de mettre du temps pour planifier les journees"
- Currently "time estimate" is optional when creating items
- Without time estimates, workload capacity calculations are meaningless
- Team must estimate time when talking to client upfront

## Tasks

### Task 1: Add Validation to Intake Form
- File: `src/components/intake/garment-services-step.tsx` (merged step from Phase 3)
- Add required validation to the time estimate field
- Prevent "Add to Order" if time is 0 or empty
- Show validation error message: "Time estimate is required for workload planning"

### Task 2: Add Validation to API Endpoint
- File: `src/app/api/intake/route.ts`
- File: `src/lib/dto.ts` (Zod schemas)
- Update the garment service schema to require `estimated_time > 0`
- Return 400 error if time estimate is missing or zero

### Task 3: Add Validation to Manage Task Modal
- File: `src/components/board/order-detail-modal.tsx` or task management modal
- If editing an existing item, ensure time estimate cannot be set to 0/empty
- Show validation on save

### Task 4: Handle Existing Data
- Check if any existing garment_services have NULL or 0 estimated_time
- These should still display correctly but show a warning indicator
- Don't retroactively block editing of old orders

### Task 5: Visual Indicator
- Mark the time field with a red asterisk (*) to indicate required
- Use consistent required field styling from the design system

## Key Files
- `src/components/intake/garment-services-step.tsx`
- `src/app/api/intake/route.ts`
- `src/lib/dto.ts`
- `src/components/board/order-detail-modal.tsx`

## Acceptance Criteria
- [ ] Cannot add item to order without entering time estimate
- [ ] Validation error appears if time is empty or 0
- [ ] API rejects orders with missing time estimates (400 error)
- [ ] Time field visually marked as required (asterisk)
- [ ] Existing orders with no time still viewable/editable
- [ ] Workload calculations always have time data for new orders

## Commit Convention
`feat: make time estimate mandatory for all new order items`
